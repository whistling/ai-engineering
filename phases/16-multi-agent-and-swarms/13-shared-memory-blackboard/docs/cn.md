# 共享内存与黑板模式

> 到 2026 年，多智能体系统中并存两种方法：**消息池**（每个人都能看到所有人的消息，像 AutoGen GroupChat 或 MetaGPT）和 **带订阅的黑板**（代理订阅相关事件，像 Context-Aware MCP 或 Matrix 框架）。两者都是多智能体系统中唯一有状态的部分——也就是说，所有有趣的 bug 都在这里。参考失效模式是 **记忆污染（memory poisoning）**：某个代理产生幻觉式“事实”，其他代理把它当作已验证的真相，准确性逐步衰减，而且比立即崩溃更难调试。本课从 stdlib 构建两种结构，注入一次中毒攻击，并展示三种在生产中真正有效的缓解措施。

**Type:** Learn + Build  
**Languages:** Python（stdlib，`threading`）  
**Prerequisites:** Phase 16 · 04（Primitive Model），Phase 16 · 09（Parallel Swarm Networks）  
**Time:** ~75 分钟

## 问题

多智能体系统需要一个共享事实的地方。一种直接的选择是“把所有东西都放在消息里传”——但这会在额外复制的同时重新发明共享状态。另一种是“给每个人一个全局日志”——但全局日志会无限增长并且容易被污染。第三种是“为每个代理投影一个视图”——可扩展，但需要严格的 schema（模式）设计。

当某个代理出现幻觉并把幻觉写入共享状态时，所有读取该状态的下游代理都会把幻觉当作事实。到人类注意到为止，推理链可能已经有五步深，而根本原因可能是第三条写入的消息。调试多智能体的准确率衰减比调试崩溃要困难得多。

这就是记忆污染。它是 MAST 分类（Cemri 等，arXiv:2503.13657）中第二常见的失败家族，并且是结构性的：任何没有溯源和不可写验证器的共享内存设计最终都会出现它。

## 概念

### 两种主要拓扑

**完整消息池（Full message pool）。** 每个代理读取每条消息。AutoGen GroupChat 和 MetaGPT 使用这种方案。简单、透明、可检查，但当代理数量超过 ~10 时不再可扩展，因为每个代理的上下文会被其他代理的工作填满。

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ 消息池         │
agent-B ──write──▶ │ (全局日志)     │ ◀──read── agent-E
                   │                │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**带订阅的黑板（Blackboard with subscription）。** 代理声明对主题的兴趣；底层只把相关消息路由给订阅者。CA-MCP（arXiv:2601.11595）和 Matrix 去中心化框架（arXiv:2511.21686）使用此方案。可扩展性更强，但需要预先设计 schema 以使订阅有意义。

```
                   ┌─ 主题：价格 ──┐
agent-A ──pub────▶ │               │ ──▶ agent-D（已订阅）
                   ├─ 主题：订单 ──┤
agent-B ──pub────▶ │               │ ──▶ agent-E（已订阅）
                   ├─ 主题：警报 ──┤
agent-C ──pub────▶ │               │ ──▶ agent-F（已订阅）
                   └────────────────┘
```

### 各自适用场景

- **完整消息池** 适合代理数量少（< 10）、角色异构、对话短期化的场景。当每个人都能看到所有内容时，追溯谁说了什么很简单。
- **黑板** 适合代理数量很多、角色相对同质但实例众多（群体）、且对话长期运行的场景。路由可以节省令牌成本并避免上下文污染。

生产系统通常混合使用：上层（规划层）使用一个小型完整消息池，下层（工作层）使用黑板。

### 记忆污染，举例说明

三个代理在做一个研究任务。Agent A 是检索代理，Agent B 是摘要生成器，Agent C 是分析师。

1. A 抓取一个页面并写入共享状态：“该研究报告了 42% 的准确率提升。”
2. 实际页面写的是“4.2% 的提升”。A 幻觉了小数点。
3. B 读取共享状态后写道：“报告称大约 42% 的准确率增长（来源：A）。"
4. C 读取共享状态后写道：“建议采用——42% 的提升是变革性的。”
5. 最终报告引用了一个从未存在过的 42% 数字。

没有代理崩溃。没有测试失败。系统“运行正常”。幻觉从某个代理的上下文跨入了共享状态，通过共享状态渗透进所有下游代理的推理链。

### 为什么这是结构性的

没有共享状态的话，A 的幻觉只停留在 A 的上下文中。下游代理会重新抓取或重新推导，可能会发现错误。使用幼稚的共享状态时，A 的上下文变成了每个人的上下文，幻觉被洗成了事实。

问题不在于共享状态本身——而在于缺乏溯源（provenance）和独立验证器（unwritable verifier）的共享状态。三种缓解措施能解决这个问题：

1. **对每次写入记录溯源。** 共享状态中的每条条目记录是谁写的、何时写的、使用了什么提示（prompt_hash）以及（如适用）引用了哪些来源。下游代理根据溯源带着怀疑去读取内容。
2. **对写入进行版本化；视为追加式（append-only）。** 更正是作为新的条目追加，替代旧条目，而不是就地修改。审计轨迹得以保留。
3. **保留至少一个不可写入共享状态的代理。** 一个只读的验证器代理会抽样条目、重新抓取引用来源并标记不一致。因为它不能写入池，所以不会被池污染。

### 黑板先例（Hayes-Roth，1985）

黑板模式比 LLM 代理早了四十年。Hayes-Roth（1985，"A Blackboard Architecture for Control"）描述了专门的 Knowledge Sources（知识源），它们观察全局黑板、贡献部分解并触发其他源。2026 年的黑板（CA-MCP、Matrix）仍是相同的模式，LLM 代理充当 Knowledge Sources，JSON blob 做为部分解。旧文献已经记录了解决写入争用、机会式控制和一致性的方案，现代系统在不断重发现这些方案。

### 投影（Projection）与全视图（Full view）

纯粹的黑板会给每个订阅者相同的投影（按主题范围）。更积极的设计是 **按代理投影（per-agent projection）**：每个代理得到一个根据其角色定制的视图。LangGraph 的状态 reducers 是 2026 年的典型实现——reducer 函数把全局状态折叠成角色特定的切片。

按代理投影进一步提高可扩展性，但需要 schema。没有 schema，你会在每个代理的提示中重新构建临时投影。

### 写入争用模式

多个代理同时写入是并发问题，不仅仅是 LLM 问题。三种常见模式有效：

- **顺序写入者（单生产者）。** 所有写入通过一个协调代理序列化。简单，但构成瓶颈。
- **带版本的乐观并发。** 每个条目带版本号；写者在版本不匹配时失败并重试。经典数据库技术。
- **主题分区。** 不同代理拥有不同主题。无跨主题争用。需要设计良好的分区边界。

多数 2026 框架默认使用顺序写入者，因为 LLM 调用本身足够慢，争用较少且瓶颈影响不大。

### 不可写验证器（Unwritable verifier）

承载量最大的缓解措施是只读验证器。实现规则：

- 验证器与团队共享状态（读取黑板或消息池）。
- 验证器没有写入共享状态的句柄——只能写入单独的验证通道。
- 验证器独立抓取写入中引用的来源，标记不一致。
- 验证器的输出被路由给人类或单独的决策代理，绝不会被送回池中。

若无此分离，验证器的输出会成为池中的新条目，受污染的池会污染验证器，进而污染验证结果。

## 构建它

`code/main.py` 用 stdlib Python 实现了两种拓扑、一个玩具中毒攻击以及三种缓解措施。

- `MessagePool` — 线程安全的追加式日志，支持完整读取。
- `Blackboard` — 基于主题的发布/订阅，支持按代理订阅。
- `ProvenanceEntry` — 每次写入都记录（writer, timestamp, prompt_hash, source_uri）。
- `PoisoningScenario` — 运行一个三代理研究任务，Agent A 幻觉小数点。打印最终报告。
- `Verifier` — 一个只读代理，重新抓取来源并标记不一致。带验证器运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行 1（无验证器）：幻觉的 42% 会传播到最终报告。
- 运行 2（有验证器）：验证器标记不一致，池被标记为“已标记（flagged）”，最终报告包含撤回说明。

## 使用方法

`outputs/skill-memory-auditor.md` 是一个技能（skill），用于审计任何多智能体系统的共享内存设计，检查溯源、版本化和验证器分离。在投入生产前，在新的多智能体架构上运行它。

## 上线建议

对于任何共享内存设计：

- 在每次写入上记录溯源信息：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志追加式（append-only）。更正作为引用被取代条目的新条目追加。
- 部署至少一个只读验证器代理，具备独立的源访问能力。
- 将验证器输出路由到单独通道，而不是回写到共享池。
- 记录“被替代写入（supersessions）”的比例——比例上升是幻觉模式早期的证据。

## 关键术语

| 术语 | 常被人如何描述 | 实际含义 |
|------|----------------|----------|
| Message pool | “共享聊天历史” | 追加式日志，所有代理读取。完全透明，但扩展性差。 |
| Blackboard | “共享工作区” | 基于主题的发布/订阅。代理订阅相关主题。可扩展性更高。 |
| Provenance | “谁写了什么” | 每次写入的元数据：写入者、时间戳、提示、引用来源。 |
| Memory poisoning | “幻觉传播” | 一个代理的错误进入共享状态，下游代理采纳为事实。 |
| Append-only | “不就地更新” | 更正作为新的条目追加并替代旧条目。保留审计轨迹。 |
| Unwritable verifier | “独立审计者” | 只读代理，重新抓取来源并标记不一致。 |
| Projection | “作用域视图” | 从全局状态计算出的按代理或按主题的视图。LangGraph 的 reducers 是典型实现。 |
| Knowledge Source | “专门代理” | Hayes-Roth 在 1985 年对黑板参与者的术语。 |
