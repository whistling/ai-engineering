# 并行 / 群体 / 网络化 架构

> 与监督者对比：没有中央决策者。代理读取共享事件总线，异步领任务，写回结果。LangGraph 明确支持用于去中心化、动态环境的“群体架构（Swarm Architecture）”。Matrix (arXiv:2511.21686) 将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除协调器瓶颈。权衡很明确：用可扩展性换取确定性和可追溯性。群体适合包含大量独立子问题的任务；不适合需要单一连贯计划的任务。

**Type:** 学习 + 构建  
**Languages:** Python（stdlib，`threading`，`queue`）  
**Prerequisites:** Phase 16 · 05 (Supervisor Pattern), Phase 16 · 04 (Primitive Model)  
**Time:** ~75 分钟

## 问题

监督者（supervisor）能扩展到少量 worker。那成百上千呢？监督者本身会成为瓶颈：关于谁做什么的每一个决策都需通过一个代理。某个缓慢的规划步骤会阻塞整个系统。

群体架构颠倒了设计。不是由中央规划者分配工作，而是工作者从共享队列中择取任务。“协调”被内置到事件总线语义中。没有协调器；系统能扩展直到队列成为限制因素。

## 概念

### 结构示意

```
                ┌──── 共享队列 ────┐
                │                   │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            结果池 (results pool)
```

没有协调器。每个工作者重复：拉取任务、处理、写回结果（并可选地入队后续任务）。

### 何时适合使用群体

- **大量独立任务。** 抓取、转换、分类等。任务之间互不依赖。  
- **任务时长可变。** 如果有些任务耗时 100ms，而有些耗时 10s，群体能自动均衡负载 —— 快速的工作者会拉取下一个任务。监督者则需要预测任务时长。  
- **追求吞吐而非严格确定性。** 关心总体完成时间，而不是严格的顺序。

### 何时不适合群体

- **有序工作流。** 如果步骤 3 需要步骤 2 的输出，群体可能导致步骤 3 在步骤 2 完成前就被执行。  
- **需要全局计划的任务。** 复杂的研究问题更适合有规划者的架构。群体中的“研究者”会产生独立事实，但难以产出连贯报告。  
- **调试困难。** 没有中央日志且工作异步，重现 bug 会很昂贵。

### Matrix (arXiv:2511.21686)

Matrix 是 2025 年的论文，把群体推到了极致：控制流和数据流都是分布式队列上的序列化消息。没有中央协调器。容错性来自消息的持久化。可扩展性成为消息代理（broker）的问题，而不是系统的问题。

贡献：一种编程模型，把多代理协调表达为“这个代理订阅哪个消息主题？”而不是“监督者下次选哪个代理？”。这让系统看起来像一个发布/订阅的事件网格。

### LangGraph 的群体架构

LangGraph 2025 文档明确把“群体架构”作为多代理模式之一：代理是节点，但边形成有向图并允许环路，任何节点都可从工作池被激活。工作者按条件从可用工作中挑选，而不是由监督者分配。

### 失效模式：饿死与热点

如果所有工作者都拉取当前最“快”的可用任务，长时间运行的任务可能永远不会被选中，直到它们成为唯一剩下的任务。这是经典的队列饿死问题。

缓解手段：
- 带显式老化的优先队列（随着等待时间增加优先级）。  
- 工作者专精化：部分工作者只处理“长”任务。  
- 背压：限制进入队列的快速任务数量。

### 与基于内容路由的关联

群体与基于内容的路由（Lesson 22）天然配对。不是通用队列，而是为每种消息类型维护一个队列。专长型工作者只订阅它们的类型。这是可扩展到数千代理的消息总线架构基础。

## 实现它

`code/main.py` 实现了一个由 4 个工作线程组成的群体，它们从共享的 `queue.Queue` 中拉取任务。任务有可变时长（有些快，有些慢）。演示对比了：

- **串行基线：** 一个工作者串行处理全部任务。  
- **固定分配：** 每个任务预先分配给特定工作者（监督者式）。  
- **群体：** 工作者从共享队列中拉取任务。

群体会自动均衡负载；固定分配在某些工作者分配到慢任务时会造成其他快速工作者空闲。

运行：

```
python3 code/main.py
```

输出显示每个工作者的任务数量（群体分配可能不均但更优）和墙钟时间。

## 使用指南

`outputs/skill-swarm-fit.md` 评估某个任务是否应使用群体或监督者。输入：任务独立性、时长方差、排序要求、可调试性需求。

## 部署要点

清单：

- **带老化的优先队列。** 防止长任务饿死。  
- **工作者幂等性。** 如果工作者在运行中崩溃，任务可能会被重复拉取。工作者必须支持幂等执行。  
- **持久化队列。** 生产环境使用 Kafka、Redis Streams 或基于数据库的队列。`queue.Queue` 仅在内存中。  
- **每任务可观测性。** 每个任务有追踪 ID；每个工作者记录该 ID 的开始/结束。  
- **背压。** 如果队列增长速度超过工作者的消耗速度，就要让生产者慢下来。

## 练习

1. 运行 `code/main.py`。在可变时长的工作负载下，群体相比串行快多少？相比固定分配快多少？  
2. 添加优先队列变体（使用 `queue.PriorityQueue`）。根据任务的“重要性”字段分配优先级。在持续负载下，观察低优先级任务是否会饿死。  
3. 实现热点检测器：当任一工作者处理的任务数量是最慢工作者的 3× 时记录日志。这表明任务时长分布有哪些信息？  
4. 阅读 Matrix 论文（arXiv:2511.21686）摘要和第 3 节。指出 Matrix 接受的一个具体权衡（例如可扩展性收益）和它放弃的一个方面（例如可追溯性、确定性）。  
5. 将群体演示改为使用包含 `(task_type, payload)` 元组的 `queue.Queue`，工作者只订阅特定类型。对于任务异构时，哪些路由规则是合理的？

## 术语表

| Term | What people say | What it actually means |
|------|----------------|------------------------|
| Swarm architecture | "Decentralized agents" | 工作者从共享队列拉取任务；没有中央协调器。 |
| Event bus | "Agents subscribe to topics" | 路由任务到工作者的消息代理，通过类型或内容进行路由。 |
| Starvation | "Task never runs" | 低优先级任务因为更高优先级工作持续到达而永远不会被选中。 |
| Hot-spotting | "One worker drowns" | 负载不均衡，某个工作者获得了大部分任务。 |
| Back-pressure | "Slow down the producer" | 当队列填满时向上游发出停止生产的信号的机制。 |
| Idempotent worker | "Safe to re-run" | 任务被处理两次也产生相同结果。必要因为工作者可能在运行中崩溃。 |
| Durable queue | "Survives crashes" | 基于磁盘或复制存储的队列；当工作者崩溃时任务不会丢失。 |
| Matrix framework | "Full message-passing swarm" | 数据流和控制流都是分布式队列上的序列化消息。 |
