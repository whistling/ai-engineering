# 检查点与回滚

> 每一次图状态转换都会被持久化。当一个 worker 崩溃时，其租约会过期，另一个 worker 会从最近的检查点继续执行。Cloudflare Durable Objects 可以在数小时或数周内保持状态。Propose-then-commit（第 15 课）为每个动作定义了回滚计划。动作后的验证闭合了整个环路。欧盟 AI 法案第 14 条对高风险系统强制要求“有效的人类监督”——在实践中，这意味着检查点必须可查询、回滚必须经过演练，并且审计轨迹必须在部署后存活。一个明显的失败模式是：如果没有幂等键和前置条件检查，瞬态故障后的重试可能会导致已批准动作被重复执行。动作后验证可以捕获这种情况。

**Type:** 学习  
**Languages:** Python (stdlib，检查点和回滚状态机)  
**Prerequisites:** Phase 15 · 12 (持久执行)，Phase 15 · 15 (Propose-then-commit)  
**Time:** ~60 分钟

## 问题

持久执行（第 12 课）使得崩溃的 agent 可以恢复。Propose-then-commit（第 15 课）使得已批准的动作可审计。本课将两者结合：当一个已批准的动作部分执行后崩溃并恢复，会发生什么？何时触发回滚，又针对什么状态回滚？

真实系统的实现方式各不相同：

- **LangGraph** 将每次图状态转换检查点写入 PostgreSQL。在 worker 崩溃时，租约释放，另一个 worker 从最近的检查点继续执行。工作流在调用 `interrupt()` 时暂停，`interrupt()` 本身也会被持久化。
- **Cloudflare Durable Objects** 可以在每个 key 级别保持数小时或数周的状态。将计算与存储（即被批准动作的状态）共址。
- **Microsoft Agent Framework** 在工作流 API 中暴露 `Checkpoint` 原语；通过重放加幂等性来覆盖重试场景。

在每种情况中，实际可行的组合是：幂等键（防止重复执行）+ 前置条件检查（确认状态仍与批准时一致）+ 动作后验证（确认副作用确实发生）+ 验证失败时回滚。

## 概念

### 每次转换都要持久化

图状态转换是指任何将工作流从一个命名状态移动到另一个命名状态的步骤。初级实现只在特定的提交点进行持久化；生产实现会持久化每一次转换。相对于可靠性提升（重放可以落在任意点、租约恢复更精确）而言，增加的写入成本（额外几次写入）是很小的代价。

### 租约恢复

当一个 worker 崩溃时，工作流不会丢失；租约（声明某个 worker 正在执行该运行的短期权利）会过期。另一个 worker 会拾取最近的检查点并恢复执行。租约机制让生产系统可以在滚动部署时保住进行中的工作。

### 幂等性加前置条件

仅有幂等性是不够的。举例：一个工作流被批准为“当余额 > $1000 时，从 A 转账 $100 到 B”。工作流已提交，执行中崩溃并随后恢复。如果只检查幂等键，恢复后重试只会执行一次（看起来是正确的）。但若在崩溃与恢复之间，A 的余额通过另一个工作流降至 $500，幂等检查仍会通过，但前置条件不再满足。如果没有前置条件检查，我们就会导致透支。

每个有影响的动作都需要两者：

- 幂等键（Idempotency key）：防止重复执行。
- 前置条件检查：确认状态仍与批准时一致。

### 动作后验证

“工具返回 200” 并不是真正的验证。真正的验证是重新读取目标状态并确认副作用确实发生。常见模式：

- 数据库更新：`UPDATE ... RETURNING *` 然后断言返回的行与预期状态一致。
- 发送邮件：提交后在已发送文件夹中检查该邮件 ID。
- 写文件：读回文件并做哈希校验。
- API 调用：在目标资源上后续执行 `GET`。

如果验证失败，工作流处于已知的坏状态。此时触发回滚。

### 回滚计划

在 propose-then-commit（第 15 课）中，每个有影响的动作都携带回滚计划。类型包括：

- 带内回滚（In-band rollback）：直接逆转副作用（在 `INSERT` 后执行 `DELETE`，发送后再发送更正邮件）。
- 补偿事务（Compensating transaction）：执行一个新的动作来中和原始动作（标准 SAGA 模式）。
- 带外回滚（Out-of-band rollback）：告警人工，暂停工作流，将坏状态留给调查。

不可逆回滚（“我们无法撤销此操作”）必须在提案中明确标注。无法回滚的动作在提交时需要更强的人工参与（第 15 课中的挑战与响应）。

### 欧盟 AI 法案第 14 条的操作性解读

第 14 条要求对高风险系统实施“有效的人类监督”。在操作层面，实施者通常这样理解：

- 检查点必须可以被审计者查询。
- 回滚必须经过演练（至少一次端到端测试）。
- 审计轨迹在部署后必须存活（检查点后端不是短暂性的）。
- 验证失败必须触发告警，而不是静默记录。

一个在提交中途崩溃、恢复后在没有验证与回滚路径的情况下完成副作用的工作流，无法通过第 14 条的考核。

### 明显的失败模式：双重执行

这类问题在生产中非常常见：

1. 动作被批准，分配幂等键 k。
2. 提交开始，执行动作，返回 200。
3. 工作流在持久化“已提交”状态之前崩溃。
4. 工作流恢复；看到“已批准但未提交”，于是重复执行。
5. 副作用发生了两次。

缓解策略：在执行前持久化一个“正在进行中的”意图，带上幂等键执行，然后仅在动作后验证成功后标记为“已提交”。如果动作已触发但状态写入失败，则在恢复时你知道需要验证并（如需要）重试触发。如果状态写入成功但动作失败，则通过恢复路径验证并确保只触发一次。

## 使用方法

`code/main.py` 实现了一个带检查点的工作流，包含幂等性、前置条件、验证与回滚。驱动程序模拟四种场景：正常运行、崩溃后的重试（幂等性防护）、前置条件失败（工作流在触发前终止）、验证失败（触发回滚）。

## 上线准备

`outputs/skill-rollback-rehearsal.md` 设计了一个针对提议工作流的回滚演练测试，并审核了检查点后端以验证审计轨迹的持久化能力。

## 练习

1. 运行 `code/main.py`。验证四种场景。对于“提交时崩溃”的情况，确认跨重试动作只触发了一次。

2. 修改“先标记为已完成，然后执行”的模式，使状态写入改为在动作之后发生。重新运行崩溃场景。测量触发了多少重复动作。

3. 为一个具体的生产动作（例如“发布到 Slack 频道”）设计回滚计划。将其分类为带内、补偿性或带外，并论证你的选择。

4. 选取一个你熟悉的工作流。识别每一次状态转换。为每个转换标注持久性需求（持久化 / 不持久化）。统计目前哪些是你没有持久化的。

5. 回滚演练测试：设计一个端到端测试，运行真实工作流、使其崩溃，然后确认回滚路径被触发。测试需要断言什么？

## 术语

| 术语 | 人们常说 | 实际含义 |
|---|---|---|
| 检查点（Checkpoint） | “保存点” | 每一次图状态转换都会被持久化到持久存储 |
| 租约（Lease） | “Worker 权利” | 声明某个 worker 正在执行该运行的短期权利；崩溃时过期 |
| 前置条件（Precondition） | “状态门控” | 断言当前状态仍与已批准的动作一致 |
| 动作后验证（Post-action verify） | “重读校验” | 在目标系统中确认副作用确实发生 |
| 带内回滚（In-band rollback） | “直接撤销” | 用逆操作直接撤销副作用 |
| 补偿事务（Compensating transaction） | “SAGA 撤销” | 通过一个新动作来中和原始动作 |
| 先标记为已完成（Mark-as-done-first） | “状态写入顺序” | 在提交返回之前先持久化已提交状态 |
| 第 14 条（Article 14） | “欧盟 AI 法案的人类监督” | 操作性要求：可查询的检查点、演练过的回滚、可审计的轨迹 |
