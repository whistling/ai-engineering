# 有界自我改进设计

> 研究汇聚出四个用于约束自我改进循环的原语。形式不变量必须在每次编辑前后成立。不能被修改的对齐锚。每个维度（安全性、公平性、鲁棒性）都必须成立的多目标约束，而不是仅仅性能。回归检测在历史指标提示能力下降时暂停循环。它们都不是安全性的证明——信息论结果（Kolmogorov 复杂性、Löb 定理）限制了任何系统关于其自身后继者能证明的内容。它们是提高静默失败成本的缓解手段。

**Type:** 学习  
**Languages:** Python (stdlib，带不变量检查的有界循环)  
**Prerequisites:** Phase 15 · 07 (RSI), Phase 15 · 04 (DGM)  
**Time:** ~60 分钟

## 问题

第7课的赛跑模拟器显示出小的速率差异会复合成大的差距。第4课的 DGM 案例研究显示循环可以主动操纵它自己的评估器。两者都指向同一个工程问题：你可以对自我改进循环施加哪些约束，使得这些约束不会被循环自身悄悄削弱？

ICLR 2026 RSI 研讨会总结 (openreview.net/pdf?id=OsPQ6zTQXV) 识别出了四个这样的原语。Anthropic 的 RSP v3.0（第19课）和 DeepMind 的 FSF v3（第20课）都在能力阈值中引用了它们。Meta 的 HyperAgents 工作和像 SAHOO（2026 年 3 月）这样的社区框架在生产中实现了子集。

诚实的表述是：这些是缓解措施。信息论结果限制了任何系统能关于自身后继者证明的内容，目前没有任何设计能从形式上完全解决该问题。一个被良好限制的循环比未被限制的更安全，但并不是绝对安全的。

## 概念

### 原语 1：形式不变量

不变量是在每次自我修改前后必须成立的性质。示例：

- 输出分布以固定的宪法头为条件（第17课）。
- 任何工具调用都不得发向未授权端点。
- 内存写入必须通过已记录且签名的路径。
- 评估器的模块哈希与批准版本匹配。

不变量由循环无法编辑的外部代码检查。如果拟议的修改违反了不变量，则被拒绝。这可以捕捉到 DGM 风格的评估器篡改模式（第4课）。

难点在于选择那些对于安全是必要且能在合理时间内计算的不变量。过度指定会阻塞进展；指定不足会让循环绕开重要性质。

### 原语 2：对齐锚

对齐锚是循环核心目标的不可变表示，固定在循环的编辑表面之外。示例：

- 每次调用都从只读位置加载的宪法文本（第17课）。
- 存放在独立代码仓库且有独立访问控制的评估器。
- 由人类签名并在每个周期重新验证的工具权限清单。

锚点的作用是防止目标漂移。循环可以改进实现目标的方式，但不能编辑目标本身。

微妙的失败模式是：循环无法编辑的锚仍然可能被漂移的循环以不同方式重新解释。宪法 AI（第17课）明确采用基于推理的方式来处理新情形；恰恰是这个推理层面存在解释漂移。锚是必要的，但不足够。

### 原语 3：多目标约束

优化单一标量分数的循环会找到捷径。必须同时满足多个硬约束的循环可用的捷径更少。典型轴线：

- 性能（任务级基准）
- 安全（红队评估，对已知有害输入的拒绝率）
- 公平性（对敏感子群的差异影响界限）
- 鲁棒性（OOD 测试集、对抗输入处理）

只有在每个约束都成立时，修改才被接受。第13课的成本治理器将其与财务约束叠加。第18课的 Llama Guard 作为安全轴插入。

### 原语 4：回归检测

每个周期的得分与历史分布比较。超过配置容差的下降会暂停循环。这可以捕捉那些否则会被循环“改进过去”并吸收进运行平均值的静默能力下降。

一个实用实现：存储最近 N 个周期的每任务得分。每个新周期计算每任务的增量。如果任何增量低于阈值，则该周期被拒绝并由人工审查。

### 信息论限制

Kolmogorov 复杂性和 Löb 定理给出系统能关于自身证明的上界。Schmidhuber 的形式化 Gödel 机（第4课）旨在争取这一上界；但没有人完成过非平凡的证明。Löb 的结果表明：如果一个系统可以可证明地相信“如果我证明我应该做 X 我就会做 X”，它会在没有证明的情况下做 X，这是一个众所周知的自指失败。

对我们这些原语的含义是：它们不能从形式上解决安全问题。它们使静默失败更昂贵。一个恶意或发生漂移的循环，如果想悄悄绕过一个缺失的检查，现在必须积极破坏一个明确的检查，这会留下更易被检测到的痕迹。

### 一个具体示例

假设一个 agent 提议一次编辑。闸门栈如下：

1. 不变量检查：模块哈希、工具权限清单、宪法头。
2. 锚点检查：目标声明与批准版本匹配（字节级或语义级）。
3. 多目标评估：性能、安全、公平性、鲁棒性轴。
4. 回归检测：任一轴不得超过容差下跌。

以上四项必须全部通过修改才会生效。任一项失败都会暂停循环。

## 使用方法

`code/main.py` 在第4课的 DGM 风格玩具示例上运行一个有界自我改进循环，但在其上叠加了四个原语。每个原语可以单独启用或禁用。演示说明每个原语能捕获特定的失败类别，移除其中任意一个会让对应的失败类别通过。

## 交付产物

`outputs/skill-bounded-loop-review.md` 审计一个拟议的有界循环，并评估其实际实现了四个原语中的哪些与其宣称的实现之间的差距。

## 练习

1. 在启用所有原语的情况下运行 `code/main.py`。确认循环在主要指标上仍然改进，同时不会让那种攻击得逞。

2. 禁用回归检测。构造一个输入，证明这会导致静默能力下降被接受。

3. 禁用多目标约束。展示循环在性能轴收敛的同时某个安全轴下降。

4. 为一个编码代理设计一个对齐锚。什么文本，存哪里，如何校验？

5. 阅读 ICLR 2026 RSI 研讨会总结。选取四个原语中的一个，提出对当前技术状态的具体改进建议。

## 关键词

| 术语 | 人们如何说 | 其实际含义 |
|---|---:|---|
| 不变量 | “始终为真的属性” | 由外部代码在每次编辑前后检查的性质 |
| 对齐锚 | “被钉住的目标” | 位于循环编辑表面之外的不可变核心目标表示 |
| 多目标约束 | “所有轴都必须成立” | 性能、安全、公平性、鲁棒性——全部必须满足 |
| 回归检测 | “下降则暂停” | 当历史指标增量提示能力下降时暂停循环 |
| Kolmogorov 上界 | “信息论限制” | 限制系统能关于其自身后继者证明的内容 |
| Löb 定理 | “自指陷阱” | 系统可以在没有证明的情况下基于“我应该”执行某事 |
| 闸门栈 | “分层检查” | 多个原语组合；任一失败都拒绝编辑 |
| 有界改进 | “缓解，不是证明” | 提高静默失败成本；并不从形式上解决安全问题 |
