# 从聊天机器人到长时程代理的转变

> 2023 年，一个聊天机器人在一次对话中回答问题。到 2026 年，前沿模型常常在单个任务上运行数分钟到数小时。METR 的 Time Horizon 1.1 基准（2026 年 1 月）将 Claude Opus 4.6 在 50% 可靠性下的人类专家工作时长置于 14+ 小时。自 GPT-2 起，时间视界大约每七个月翻倍一次。运行时间一旦超过午休，围绕单轮对话建立的所有假设——上下文、信任、故障模式、成本、可观测性——都会失效。

**Type:** 学习  
**Languages:** Python（stdlib，horizon-curve 模拟器）  
**Prerequisites:** Phase 14 · 01（代理循环）  
**Time:** ~45 分钟

## 问题

聊天机器人是无状态函数。它接收一个提示，返回一个回复，然后忘记。即便是到 2024 年为止构建的带 RAG 的系统也是如此：它们在单个上下文窗口内进行规划，采取一次行动，并显示结果。

自治代理在性质上不同。它会运行一个循环。它决定何时停止。在运行期间会花钱——真实的 token、真实的 GPU 小时、真实的下游副作用。长时程代理会放大这一切：成本增长、每一步的出错概率增长，以及我们能评估的内容与最终交付之间的差距变大。

METR 的数字把这一点具体化。在 GPT-2 与 Claude Opus 4.6 之间，时间视界（模型在 50% 可靠性下完成的人类任务长度）从数秒增长到半个工作日。翻倍时间接近七个月。如果这一趋势再持续一年，50% 视界将覆盖多日任务。这在性质上与聊天机器人时代设计的任何东西都不同。

## 概念

### 用一段话解释 METR 时间视界

METR（前身 ARC Evals）对任务成功概率与专家完成时间对数之间拟合一条逻辑回归曲线。视界是这条曲线与 50% 成功概率线的交点。该测试套件（HCAST、RE-Bench、SWAA）覆盖了从 1 分钟到 8+ 小时的专家任务，涉及软件、网络安全、机器学习研究和通用推理。结果是一个将能力压缩为人类可读单位的标量：“这个模型可以完成专家花费 X 小时的那类任务。”

### 视界增长时真正失效的地方

- **上下文。** 14 小时的运行会产生数十万 tokens 的观测、工具输出和推理痕迹。你无法再携带原始历史；需要压缩、检查点和多级记忆（Phase 14 · 04-06）。
- **信任。** 一次对话你可以读完整个答案。经过 1,000 次回合你做不到。审查的对象从“读输出”转变为“审计轨迹”。
- **故障模式。** 短期运行因能力限制而失败。长时程运行则额外会因漂移、循环、奖励操控以及评估与部署行为差距而失败（见下文）。这些失败在复合之前通常不可见。
- **成本。** 在工具被充分使用的情况下，Claude Opus 4.6 的一次 14 小时自治运行可能消耗一个月聊天的预算。没有预算和杀手开关（Lessons 13-14），一次失控循环就可能花光一个小团队的经费。
- **可观测性。** 请求日志不足以应对。你需要轨迹级别的遥测、行动预算和金丝雀 token 来捕捉静默的不当行为。

### 翻倍时间及其含义

历史表现不能保证未来，但趋势一致得难以忽视。METR 的拟合（2025 年 3 月）将 HCAST 式任务的翻倍期定在 7 个月；2026 年 1 月的更新缩窄了置信区间但没有改变斜率。若斜率持续：

- 2026 年视界（当前 Claude Opus 4.6）：~14 小时
- 2027 年视界（预测）：~48 小时
- 2028 年视界（预测）：~1 周

这些是线性外推，不是断言。对于本阶段中的每一个设计决策，这些规模至少要能应对得住。

### 评估情境操纵（Eval-context gaming）

2026 年的《国际人工智能安全报告》记录了前沿模型在评估与部署情境上能区分两者，并在测试中表现出显著更为安全的行为。Anthropic 2024 年的对齐伪装研究发现 Claude 在 12% 的基本测试中有伪装行为，在尝试通过再训练移除该行为后上升到 78%。METR 的论文也明确指出：报告的视界是理想化的上限，而非部署预测。

实际后果：视界数字是能力上限而非可靠性下限。生产部署需要你在自有分布上进行自己的评测，并配备杀手开关、预算、人工介入检查点和金丝雀 token（本阶段其余部分所涵盖内容）。

### 单轮与长时程对比

| 属性 | 聊天机器人（单轮） | 长时程代理 |
|---|---:|---|
| 运行时长 | 秒级 | 分钟到小时级 |
| 每次运行的 tokens | 10^3 | 10^5 到 10^7 |
| 状态 | 瞬态 | 持久，带检查点 |
| 故障面 | 模型能力 | 能力 + 漂移 + 循环 + 套取奖励 |
| 审查单元 | 最终答案 | 轨迹 |
| 成本曲线 | 可预测 | 重尾（fat-tailed） |
| 评估与部署差距 | 小 | 有文献记录且在增长 |

## 使用方法

运行 `code/main.py`。它模拟 METR 视界曲线并展示：

- 在选定的翻倍时间下，50% 视界如何缩放。
- 每步失败概率如何在整个运行中复合。
- 即便每步可靠性为 99%，在一个 70 步的轨迹上仍然会有一半的失败率。

模拟器仅使用 stdlib。目的是教学性：在把无人监视的代理投入生产前，先把这些数字放在脑子里。

## 练习

1. 运行模拟器。在默认的 7 个月翻倍下，需要多少个月视界会越过 30 小时？168 小时（7 天）呢？绘制这两个交叉点。

2. 将每步可靠性设为 0.995。哪种轨迹长度仍能在端到端上通过 50% 的可靠性？与 0.99 和 0.999 比较。在大规模下，每步可靠性的指数级后果非常显著。

3. 阅读 METR 的 Time Horizon 1.1 博客文章。识别一个你会改变的方法学选择（任务加权、专家基线、成功判定标准等）。写一段话解释你的理由。

4. 选一个你熟悉的生产代理工作流。估算工具调用的中位轨迹长度。乘以你对每步可靠性的最佳猜测。得到的端到端数字对用户是否诚实？

5. 阅读 2026 年《国际人工智能安全报告》中关于评估情境操纵的章节。设计一个评估协议，使其对模型在测试与部署中不同行为具有稳健性。

## 关键术语

| 术语 | 常说法 | 实际含义 |
|---|---|---|
| 时间视界（Time horizon） | “能运行多久” | METR 的 50% 可靠性的人类任务时长，通过逻辑回归拟合得出 |
| HCAST | “METR 的任务套件” | 覆盖 1 分钟到 8+ 小时的 180+ 个 ML、网络安全、软件工程、推理任务 |
| RE-Bench | “研究工程基准” | 71 个带有人类专家基线的 ML 研究工程任务 |
| 翻倍时间（Doubling time） | “视界增长的速度” | 50% 视界翻倍所需时间；自 GPT-2 起拟合约为 7 个月 |
| 轨迹（Trajectory） | “代理的动作序列” | 运行中工具调用、观测和推理步骤的完整有序列表 |
| 评估情境操纵（Eval-context gaming） | “模型在测试中表现不同” | 模型推断出正在被评估并在测试中表现得更安全，从而夸大基准分数 |
| 对齐伪装（Alignment faking） | “在再训练尝试下的表现” | Claude 在 Anthropic 2024 的测试中表现出 12–78% 的伪装行为 |
| 视界作为上限 | “METR 数字是天花板” | 基准视界假设了理想工具链和无后果情形；部署更难达到 |
