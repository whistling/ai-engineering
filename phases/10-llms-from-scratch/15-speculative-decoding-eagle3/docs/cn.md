# 投机性解码与 EAGLE-3

> 第 7 阶段 · 第 16 课证明了数学结论：Leviathan 拒绝规则精确保留验证器的分布。本课从训练栈的角度讲述 2026 年生产级投机性解码。EAGLE-3 将草稿模型从廉价近似变成了一个为此目的构建的小型网络，直接在验证器的隐藏状态上训练，并加入了一个在训练时的测试循环以对齐训练与推理分布。结果：端到端加速 3× 到 6.5×，聊天场景下每标记接受率超过 0.9，且无分布性折衷。到 2026 年，每个生产推理栈默认都集成此方案。

- 从接受率 `α` 和 草稿到验证器的成本比 `c` 计算期望加速，针对不同情形选择最优草稿长度 `N`。  
- 从头实现完整的投机性循环：生成草稿、并行验证、按位置拒绝、从残差分布重采样、在拒绝时回滚 KV 缓存，以及在完全接受时发出奖励令牌。

## 问题背景

在 H100 上，对一个 70B 模型进行自回归解码速度可能只有约 35 tokens/s。GPU 并未被计算单元饱和，瓶颈在内存带宽：每个 Token 都要从 HBM 加载 70B 的权重，做一次算术运算，输出一个浮点数，计算单元大多处于空闲状态。

投机性解码将其转化为一个可实际解决的吞吐量问题。一个廉价的草稿模型通过 N 次小的前向计算提出 N 个令牌。验证器在前缀加上所有 N 个草稿上做一次前向，得到多个位置的分布。如果在位置 i 上验证器的分布在统计意义上与草稿一致，我们接受；否则拒绝并从残差分布中采样修正。一次大模型前向可以产生高达 N+1 个被接受的令牌，而不是仅一个。

关键定理是 Leviathan, Kalman, Matias (ICML 2023)：输出分布与直接从验证器采样所得的分布**完全相同**。不是近似，而是完全相同。这正是投机性解码可用于生产的根本原因——它只是一个纯粹的延迟优化，没有质量折衷。

第 7 阶段 · 第 16 课给出了数学证明。本课给出训练栈上的工程实现。更好的草稿比单纯更便宜的草稿能带来约 2× 的额外加速。EAGLE、EAGLE-2 与 EAGLE-3（Li 等，2024–2025）将“草稿 = 同族模型的更小版本”发展成一门精确的工程学。到 2026 年，生产推理服务器默认使用 EAGLE-3。

## 概念

### 不变量：Leviathan 拒绝采样

设 `p(t)` 是在给定某个前缀时草稿对下一个令牌的分布，`q(t)` 是验证器的分布。从 `p` 中采样令牌 `d ~ p`。以概率 `min(1, q(d) / p(d))` 接受它；在拒绝时，从残差分布 `(q - p)_+ / ||(q - p)_+||_1` 中采样。由此得到的样本按 `q` 分布。这一结论与 `p` 的好坏无关——`p` 越差，拒绝越频繁，但输出仍然是精确的。

将 N 个此类调用串联起来，使用对 `prefix + d_1 + ... + d_N` 的一次验证前向。验证器同时返回 `q_1, q_2, ..., q_{N+1}`。从左到右处理：在第 j 个位置首次拒绝时，从 `residual(q_j, p_j)` 中采样修正并停止；如果全部接受，则从 `q_{N+1}` 中采样一个奖励令牌。

### 影响加速的因素

令 `α` 为每个草稿令牌的期望接受率，令 `c = cost(draft) / cost(verifier)` 为成本比。每次验证前向的期望被接受令牌数为：

```
E[accepted] = (1 - α^(N+1)) / (1 - α)
```

每个被接受令牌的期望总时间为 `(N * c + 1) / E[accepted]`。对 `N` 最小化这个量即可得到最优点。对于 `α = 0.8, c = 0.05`：最优 `N` 约为 5–7，加速约 3.2×。对于 `α = 0.95, c = 0.02`：最优 `N` 约为 8–10，加速可达 5×。

最大的杠杆是 `α`。在固定 `N = 5` 时，将 `α` 从 0.6（原始草稿）提升到 0.9（EAGLE-3），每次验证前向的期望接受令牌数从 2.2 提升到 4.1。相同验证器下几乎带来 2× 的吞吐提升。

### 两年演进概览

**原始投机性（Leviathan，2023）。** 草稿模型是同家族里独立训练的更小 LLM。接入简单，`α ≈ 0.6`，最好可得约 2× 加速。

**EAGLE-1（Li 等，2024）。** 草稿为一个极小的 transformer（通常 1–2 层），以验证器的最后一层隐藏态为输入直接预测下一个令牌。由于草稿可见验证器的特征表示，其分布更接近验证器，`α` 提升至 0.7–0.8。

**EAGLE-2（Li 等，2024）。** 增加了动态草稿树：不是只提出一条长度为 N 的序列，而是提出一棵候选小树，在一次验证前向中对每个节点评分（树注意力），然后沿最高概率路径行走。草稿长度对每步自适应，按被接受路径计的 `α` 上升到 0.85 以上。

**EAGLE-3（Li 等，2025，NeurIPS）。** 额外两项改进。第一，放弃特征预测损失——EAGLE-1/2 将草稿训练为匹配验证器的隐藏态，这限制了数据的作用。EAGLE-3 直接用于令牌预测。第二，在训练时做测试（TTT）：在草稿训练中，将草稿自己的先前预测作为输入反馈多步运行，和推理时的运行方式保持一致，从而对齐训练与测试分布，停止误差累积。实测加速：聊天场景最高可达 6.5×；在 H100 上 SGLang、batch 64 情形下吞吐量提升约 38%。

### KV 缓存回滚

一次验证会将验证器的 KV 缓存延伸 N 条目。如果在位置 `j` 拒绝，缓存中超过位置 `j-1` 的内容就不再正确。常见实现有两种：写入临时缓冲区并在接受时提交（例如 vLLM、TensorRT-LLM），或保留物理 KV 缓存并用逻辑长度截断（在拒绝时回退）。无论哪种，回滚的成本是每层每头的字节数，相较于前向成本可以忽略不计。

对于 EAGLE-2 的树搜索，验证器使用一个遵守树拓扑的非因果掩码做注意力。工程上有些繁琐，但计算上是带自定义掩码的标准 flash-attention 调用。

### 2026 年的草稿架构

| 策略 | 草稿类型 | `α` | 加速 | 训练成本 |
|------|----------|-----:|------:|---------:|
| Vanilla | 独立小型 LLM | 0.55–0.70 | 1.8–2.3× | 无（重用已有小模型） |
| Medusa | 在验证器上加多头 LM | 0.65–0.75 | 2–3× | ~10 亿 SFT tokens |
| EAGLE-1 | 在隐藏态上的 1 层 transformer | 0.70–0.80 | 2.5–3× | ~600 亿 tokens |
| EAGLE-2 | EAGLE-1 + 动态草稿树 | 0.80–0.88 | 3–4× | ~600 亿 tokens |
| EAGLE-3 | 多层特征融合 + TTT | 0.88–0.92 | 3.5–6.5× | ~600–2000 亿 tokens |
| Lookahead | 无草稿（Jacobi 迭代） | N/A | 1.3–1.6× | 无 |

在 2026 年生产环境：vLLM 与 SGLang 在可用时默认使用 EAGLE-3，否则使用 EAGLE-2。TensorRT-LLM 对于 Meta 与 NVIDIA 的公开模型提供最快的 Medusa 路径。llama.cpp 在 CPU 部署上通常使用原始草稿（vanilla draft）。

## 实现

参见 `code/main.py`。这是完整的 Leviathan 投机性循环，包含所有部分：长度为 N 的草稿、验证器并行前向、逐位置拒绝、残差采样、奖励令牌、KV 回滚，以及对输出分布与直接从 `q` 采样的一致性进行经验验证。

### 步骤 1：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 步骤 2：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 步骤 3：完整的投机性步骤

`spec_step` 函数从 `p` 中生成 N 个草稿令牌，然后对所有这些令牌一次性并行地评估 `q`。对每个草稿令牌应用拒绝规则，在第一个拒绝处从残差中采样修正并停止。如果所有草稿都被接受，则从 `q_{N+1}` 中发出一个奖励令牌。

### 步骤 4：KV 回滚的记账

模拟器为每个工作线程跟踪一个逻辑的 `kv_length`。在接受 k 个草稿后，`kv_length += k`。在位置 `j` 处拒绝时，缓存已经写入了超过 `j` 的内容，但逻辑长度设为 `prefix_length + j + 1` —— 即修正令牌之后的一位。后续读取会以逻辑长度为界进行截断。

### 步骤 5：Leviathan 检验

运行 50,000 次投机性步骤。统计被接受令牌的经验分布。与从 `q` 直接采样的 50,000 次结果进行比较。卡方统计量应远低于临界值。实践中定理通过检验。

### 步骤 6：关于 α 的加速曲线

通过在不同振幅下将 `p` 从 `q` 扰动来扫描草稿质量。测量 `α`，然后绘制 `α` 与 `N` 下每次验证调用的期望令牌数。代码会打印表格，展示 EAGLE-3 级别的草稿质量（`α ≈ 0.9`）如何解锁每次验证调用 4–5 个令牌。

## 使用场景

生产级的 `vllm serve` 与 EAGLE-3 的示例：

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 5,
    "method": "eagle3"
  }'
```

在 H100、batch 64 的 SGLang 上使用 EAGLE-3：与 batch-64 的原始解码相比，吞吐量大约提升 1.38×（依据 EAGLE-3 论文）。

何时使用投机性解码：

- 任何以 p50 延迟为关键的交互式聊天负载。  
- 代码生成与结构化输出（JSON、SQL）。由于目标分布高度可预测，`α` 常超过 0.9。  
- 长文本生成（数千令牌）。摊销后的加速持续有效。

不适合使用的情况：

- 非常小的模型（< 3B）。草稿相对验证器并不便宜。  
- 极小的 batch-1 CPU 部署。草稿模型的内存开销可能不值得。  
- 非常高温度的创造性采样场景，`α` 会崩塌。

## 关键词

| 术语 | 常用说法 | 实际含义 |
|------|---------|---------|
| Leviathan rule | “min(1, q over p)” | 伯努利接受/拒绝，接受概率为 `min(1, q(d)/p(d))`；在拒绝时从残差采样，精确保留验证器分布 |
| Residual distribution | “(q minus p) plus, normalized” | `(q - p)_+` 在 0 处截断并重新归一化——拒绝时应采样的正确分布 |
| Acceptance rate α | “草稿有多常正确” | 在拒绝规则下的每标记伯努利成功的期望概率；决定所有加速数学 |
| EAGLE-1 | “隐藏态草稿” | 以验证器最后一层隐藏态为条件的小型 transformer 草稿（Li 等，2024） |
| EAGLE-2 | “动态草稿树” | EAGLE-1 加上在一次验证前向中评分的候选树 |
| EAGLE-3 | “训练时测试（TTT）” | 放弃特征预测损失，直接训练令牌预测，并在训练中将草稿喂回其自身输出 |
| Training-time test (TTT) | “暴露偏差修复” | 在训练中以自回归方式运行草稿，使训练与测试的输入分布一致——是 scheduled sampling 的直接对应 |
| KV rollback | “撤销被拒绝的草稿” | 将验证器的 KV 缓存重置为被接受前缀长度的记账操作 |
| Bonus token | “免费那一个” | 当所有 `N` 个草稿都被接受时，从 `q_{N+1}` 中采样的额外令牌，零额外验证器成本 |
| Tree attention | “一次验证多个候选” | 使用遵守草稿树拓扑的非因果掩码的注意力；在一次前向内计算树中每个节点的 `q_i` |
