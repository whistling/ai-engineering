# SRE for AI — 多代理事件响应、运行手册、预测性检测

> AI SRE 使用基于基础设施数据（日志、运行手册、服务拓扑）的 LLM，通过 RAG 自动化调查、文档和协调阶段。到 2026 年的架构模式是多代理编排 — 专门化代理（日志、指标、运行手册）由一个监督者协调；AI 提出假设和查询，人类对判断性决策进行审批。Datadog Bits AI 和 Azure SRE Agent 以托管产品形式交付。运行手册在演进：NeuBird Hawkeye 使用对抗性评估（两个模型独立分析同一事件；一致 = 置信，不一致 = 不确定）；操作记忆跨团队更替持续存在。自动修复保持谨慎：AI 建议，人类审批。完全自主的动作范围狭窄（重启 Pod、回滚特定部署）并有严格的护栏 — 任何宣称“设定后忘记”的卖家都在过度宣传。新兴前沿：事件前预测。MIT 的研究报告称，一个在历史日志 + GPU 温度 + API 错误模式上训练的 LLM 在测试集上能提前 10–15 分钟预测 89% 的故障。预测：到 2026 年底，95% 的企业级 LLM 将实现自动故障转移。

**Type:** 学习  
**Languages:** Python（stdlib，玩具多代理事件分诊模拟器）  
**Prerequisites:** Phase 17 · 13（可观测性），Phase 17 · 24（混沌工程）  
**Time:** ~60 分钟

## 学习目标

- 绘制多代理 AI SRE 架构图：监督者 + 专门化代理（日志、指标、运行手册）+ 人类审批门。
- 解释为什么自动修复范围应当狭窄（重启 Pod、回滚部署）而非广泛（重构服务）。
- 说出对抗性评估模式名称（NeuBird Hawkeye）：两个模型一致 = 置信；不一致 = 升级。
- 引用 MIT 的 89% 早期检测结果以及操作约束：没有执行能力的预测只是仪表板。

## 问题背景

值班工程师在凌晨 3 点被告警：“结账系统错误率高”。他们查看 Datadog、Loki、三份运行手册和部署日志。30 分钟后发现根因是 vLLM 因 KV 缓存激增导致的 OOM。他们重启了 Pod；错误消失。

到 2026 年，这次调查的前 20 分钟大部分是可自动化的。按服务分组日志、与最近部署相关联、与运行手册匹配 — 这些都是 RAG + 工具使用的范式。一个监督型代理可以做第一轮分诊并在人工打开 Datadog 之前提出假设。

完全自主修复是另一个问题。重启 Pod：安全。扩展 GPU 池：如果策略允许则安全。重构服务：绝对不行。纪律在于划定这条狭窄的界限。

## 概念

### 多代理架构

```
          事件
             │
             ▼
        监督者
        /    |    \
       ▼     ▼     ▼
  日志代理  指标代理  运行手册代理
       │     │     │
       └─────┴─────┘
             │
             ▼
        假设 + 证据
             │
             ▼
        人类审批
             │
             ▼
        行动（有限集）
```

监督者将事件拆分为子查询。专门化代理拥有工具访问权限（日志检索、PromQL、文档检索）。监督者合成结果，向人类呈现假设 + 证据。人类进行批准或重定向。

### 自动修复范围

**安全（狭窄）**：重启 Pod、回滚特定部署、在预批准范围内扩缩资源池、启用预批准的 feature flag。

**不安全（广泛）**：改变服务拓扑、修改资源限制、部署新代码、更改 IAM、修改数据库结构。

任何宣称“设定后忘记”的说法都是过度宣传。随着 AI SRE 的成熟，安全集会扩展，但边界是真实存在的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果它们在根因上达成一致，置信度高；如果不一致，则向人类升级，并同时展示两种假设。简单的模式，有效过滤出虚构的根因。

### 操作记忆

团队人员更替是传统 SRE 的隐形杀手 — 部落知识会流失。AI SRE 将运行手册 + 事后分析存储在向量数据库；代理在每次新事件中检索。当新工程师加入时，AI 已具备完整历史。

### 事件前预测

MIT 2025 年研究：一个在历史日志、GPU 温度和 API 错误模式上训练的 LLM 在测试集上能提前 10–15 分钟预测 89% 的故障。

现实检验：没有执行力的预测只是仪表板。操作问题是“当我们预测到时，我们做什么？”提前 drain？告警？自动扩容？答案取决于策略。

### 2026 年的产品

- **Datadog Bits AI** — Datadog 内的托管 SRE 助手。  
- **Azure SRE Agent** — Azure 原生方案。  
- **NeuBird Hawkeye** — 对抗性评估 + 操作记忆。  
- **PagerDuty AIOps** — 分诊 + 去重。  
- **Incident.io Autopilot** — 事故指挥 + 协调。

### 将运行手册作为代码

运行手册正从 Confluence 页面演进为带结构的版本化 Markdown（结构化部分：症状、假设、验证、操作）。结构化运行手册更利于 RAG 检索。任何 AI-SRE 推广的起点都是把非结构化的运行手册转成结构化格式。

### 你应该记住的数据

- MIT 早期检测：89% 的故障，提前 10–15 分钟。  
- 多代理分诊：监督者 +（日志、指标、运行手册）+ 人类。  
- 安全自动修复集：重启 Pod、回滚部署、在界限内扩缩。  
- 对抗性评估：两个模型独立；一致 = 置信。

## 练习

1. 运行 `code/main.py`。如果日志代理和指标代理意见不一致，监督者如何决策？  
2. 为你的服务定义三个“安全”自动修复动作。为每一个给出理由。  
3. 编写一个结构化运行手册模板：章节、必填字段、验证命令。  
4. 预测检测在 12 分钟前触发。你的策略是什么 — 告警、预排空（pre-drain），还是两者兼顾？  
5. 论证一个 3 人团队在 2026 年是否应该采用 AI SRE 或等待。考虑成熟度、事件量、风险。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| AI SRE | “值班的代理” | 基于 LLM 的事件调查 + 协调 |
| Supervisor agent | “编排者” | 把事件拆分为子查询的顶层代理 |
| Specialized agent | “领域代理” | 拥有工具访问权限的子代理（日志、指标、运行手册） |

| Operational memory | “向量运行手册” | 存在向量数据库中的事后分析与运行手册，用于 RAG |
| Adversarial eval | “双模型检查” | 独立分析；一致 = 置信 |
| NeuBird Hawkeye | “对抗性方案” | 具有对抗性评估 + 记忆模式的产品 |
