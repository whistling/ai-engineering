# Long-Video Understanding at Million-Token Context

> 一小时的 4K 视频在 24 FPS、打补丁并嵌入后，会产生约数千万级别的令牌。两小时的播客转录是 30,000 令牌。即便经过激进的池化压缩，一部完整的蓝光电影也有数十万令牌。Google 的 Gemini 1.5（2024 年 3 月）以 1000 万令牌的上下文开启了这个时代，能够在数小时的视频中做出可靠的“大海捞针”级别回忆。LWM（Liu 等，2024 年 2 月）展示了 ring attention 的扩展路径。LongVILA 和 Video-XL 在摄取规模上进一步扩展。VideoAgent 则用 agentic retrieval（代理检索）替换了原始上下文。每种方法在计算、检索率和工程复杂度上有不同的权衡。本课把它们并列比较。

**Type:** 构建  
**Languages:** Python（stdlib、needle-in-haystack 模拟器 + agentic-retrieval 路由器）  
**Prerequisites:** 阶段 12 · 17（视频时间令牌）  
**Time:** ~180 分钟

## 学习目标

- 计算不同 FPS 和池化策略下，长视频的总视觉令牌数。
- 解释三条扩展路径：暴力上下文（Gemini 1.5）、ring attention（LWM）、令牌压缩（LongVILA / Video-XL）。
- 比较原始上下文视频 VLM 与 agentic-retrieval 视频 VLM（VideoAgent）在准确率与延迟上的差异。
- 设计一个针对 30 分钟视频的“大海捞针”测试，并在特定分钟测量召回率。

## 问题背景

一帧大小等同于 Qwen2.5-VL 的补丁（原生 384 分辨率）约为 729 个令牌。采用 3x3 池化后每帧为 81 个令牌。30 分钟、1 FPS = 1800 帧 = 145,800 个令牌。对 2025 年的开源 VLM 来说是可行但紧张的。2 FPS 时为 291,600 个令牌——只有最大上下文模型能容纳。

2 小时电影、1 FPS 为 583k 令牌。超出多数 2026 年开源模型；需要 Gemini 2.5 Pro 或更激进的池化。

出现了三条（+1）扩展路径。

## 概念

### 路径一：暴力上下文（Gemini 1.5、Claude Opus）

用硬件解决问题。将上下文扩展到数百万令牌，在一次前向传播中处理所有内容。

Gemini 1.5 Pro 推出时支持 1M 令牌；Gemini 1.5 Ultra 到 10M；2026 年的 Gemini 2.5 Pro 能可靠处理数小时视频。论文（arXiv:2403.05530）记录了在 ~9.5M 令牌范围内达到 99.7% 的大海捞针召回率。

工程实现：自定义注意力实现 + 内存层次结构（局部 + 全局 + 稀疏）以及用于长上下文效率的 MoE 专家路由。未全部公开开源。

### 路径二：Ring attention（LWM、LongVILA）

Ring attention 将长序列分布到多设备的“环”上，每个设备保存一段。跨序列的注意力通过每个设备按环形发送其数据块、计算部分注意力并聚合来完成。

LWM（Liu 等，2024）用这种方式训练了 1M 令牌上下文模型。训练计算随上下文线性扩展，而非二次增长——注意力的二次复杂度被在环上设备间摊销。

LongVILA（arXiv:2408.10188）将该模式适配到 VLM。1400 帧、每帧 192 令牌 = 268k 上下文，使用 8 路并行的 ring attention 训练。

### 路径三：令牌压缩（Video-XL、LongVA）

比暴力上下文便宜：在 LLM 看到序列前做激进压缩。

Video-XL（arXiv:2409.14485）使用可视汇总令牌：每 N 帧生成一个“汇总”令牌，汇总令牌对该 N 帧做注意力。在推理时，LLM 只看到每个片段的一个汇总令牌，从而大幅缩小上下文。

LongVA 通过“长上下文迁移”技术将 LLM 上下文从 200k 扩展到 2M，并通过共享表示把长文本上下文迁移到长视频上。

令牌压缩用可扩展性换取对精确时间戳的召回：模型能大致知道发生了什么，但有时会错过确切帧。

### 路径四：Agentic retrieval（VideoAgent）

不要把整个视频都喂给 LLM。把视频当作数据库，用 LLM 来查询它。

VideoAgent（arXiv:2403.10517）流程：

1. LLM 读取问题。  
2. LLM 向检索工具请求相关片段（“显示包含猫的片段”）。  
3. 工具返回匹配的时间戳片段。  
4. LLM 通过 VLM 阅读这些片段。  
5. LLM 生成答案或提出后续查询。

这是将 LLM-as-agent 模式应用于长视频的方案。推理更便宜（只编码相关片段），工程更难（检索质量成为瓶颈）。

### 大海捞针基准

标准的长上下文测试：在视频随机位置插入一个唯一的视觉或文本标记，然后提出需要回忆该标记的查询。

度量：跨视频长度与标记位置的 Recall@k。

Gemini 2.5 Pro 在长达 90 分钟的视频上召回率 >99%。开源的 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟时约 85–90%，超过 60 分钟后退化。

VideoAgent 如果检索工具足够好，可以在 2 小时以上场景中匹配或超过原始上下文模型，因为检索能直接命中针。

### 如何选择路径

- 对于 15 分钟以内、追求前沿准确率：开源 72B + 原生上下文通常可行。推荐 Qwen2.5-VL-72B。  
- 对于 30 分钟到 1 小时：开源方向选 LongVILA 或 Video-XL；闭源可选 Gemini 2.5 Pro。质量门槛高则倾向闭源。  
- 对于 2 小时以上：VideoAgent 或类似的检索模式更合适。或将视频分层汇总后再处理（先摘要再逐层展开）。

### 2026 年生产模式

实际生产中的长视频流水线通常是混合式：

1. 对整个视频运行动态 FPS 抽样 + 激进池化（得到 ~100k 令牌的全局表示）。  
2. 交给 72B VLM 做全局摘要。  
3. 若用户问细节，使用摘要作为索引运行 agentic retrieval。

这种组合同时利用了暴力上下文做全局理解、检索做局部细节。

## 使用方法

`code/main.py`:

- 计算从 1 分钟到 3 小时、不同 FPS + 池化下的视频令牌预算。  
- 模拟一次大海捞针运行：在随机时间戳注入标记、提出问题、评估召回。  
- 包含一个 agentic-retrieval 路由器模拟器，选择要送到下游 VLM 的特定片段。

运行预算表，感受规模差距。

## 术语表

| 术语 | 俗称 | 实际含义 |
|------|------|----------|
| Brute context | “更多令牌” | 将 LLM 上下文扩展到数百万令牌；在一次前向传播中处理所有内容 |
| Ring attention | “LWM 风格并行” | 分布式注意力模式，每个设备保存一段并按环形旋转计算注意力 |
| Token compression | “汇总令牌” | 在 LLM 之前通过学习型压缩器减少每个片段的令牌数 |
| 大海捞针测试 | “NIH test” | 在随机点插入唯一标记，测试时要求模型回忆该标记 |
| Agentic retrieval | “LLM 作为查询计划器” | LLM 向检索工具请求相关片段，读取这些片段后生成答案 |
| VideoAgent | “视频的检索模式” | 经典的 agentic-retrieval 设计：问题 -> 工具 -> 片段 -> 答案 |
