# 混合记忆：向量 + 图 + KV (Mem0)

> Mem0（Chhikara 等，2025）将记忆视为三种并行存储 —— 向量用于语义相似性，KV 用于快速事实查找，图用于实体关系推理。一个评分层在检索时将三者融合。到 2026 年，这是外部记忆的生产标准。

**Type:** 构建  
**Languages:** Python（stdlib）  
**Prerequisites:** Phase 14 · 07 (MemGPT), Phase 14 · 08 (Letta Blocks)  
**Time:** ~75 分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅 KV）对于智能体记忆是不足够的。
- 说出 Mem0 的三种并行存储以及每种存储优化的目标。
- 描述 Mem0 的融合评分 —— 相关性、重要性、时效性（recency） —— 以及为什么它是加权和，而不是层级。
- 在 stdlib 中实现一个玩具三存储记忆，包含一个在三处写入的 `add()` 和一个融合结果的 `search()` 实现。

## 问题背景

单一存储对以下三类查询总有一类错过：

- 语义相似性 — “上周我们讨论过关于智能体漂移的哪些内容？” 向量胜出；KV 与图会漏掉。
- 事实查找 — “用户的电话号码是什么？” KV 胜出；向量浪费资源，图则过度。
- 关系推理 — “哪些客户属于相同的计费实体？” 图胜出；向量和 KV 无法回答。

生产环境中的智能体会在一次会话中发出这三类查询。单一存储总会对其中两类出错。Mem0 的贡献是将三者通过统一的 `add`/`search` 接口连接起来，并用一个融合评分函数将结果合并。

## 概念

### 三个并行存储

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text, user_id, metadata)` 时：

1. 从文本中提取候选事实（LLM 驱动的步骤）。
2. 将每个事实写入向量存储（嵌入）以用于语义检索。
3. 将每个事实写入 KV 存储，键为 `(user_id, fact_type, entity)`，以实现 O(1) 查找。
4. 将每个事实作为带类型的边写入图存储（Mem0g），用于关系查询。

在 `search(query, user_id)` 时：

1. 向量存储按嵌入余弦相似度返回 top-k。
2. KV 存储按从查询推导出的 `(user_id, type, entity)` 返回直接命中。
3. 图存储返回从查询实体可达的子图。
4. 一个评分层将三者融合。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- 相关性（Relevance）—— 向量使用余弦，KV 使用精确匹配，图使用路径权重。
- 重要性（Importance）—— 在写入时打标签或通过学习获得（有些事实更重要：姓名、ID、策略）。
- 时效性（Recency）—— 自上次写入或读取以来的指数衰减。

权重根据产品调优。聊天类智能体偏高 `w_recency`；合规模型偏高 `w_importance`；检索型偏高 `w_relevance`。

### Mem0g 与时间推理

Mem0g 添加了冲突检测。当新事实与现有边冲突时，现有边被标记为无效但不删除。时序查询（“三月份用户的城市是什么？”）遍历在该时间点有效的子图。

这是 Letta 的失效模式在合规级别上的通用化实现。

### 基准数据

Mem0 论文报告（2025）：

- LoCoMo（长篇对话记忆）：91.6
- LongMemEval（长时间范围情节记忆）：93.4
- BEAM 1M（1M-token 记忆基准）：64.1

比较基线（全上下文 128k LLM、平面向量存储、平面 KV）都输 10+ 分。仅凭基准并不能完全说明选择理由 —— 运行形态更重要 —— 但这些数字显示融合设计不是误差范围内的改进。

### 作用域分类

Mem0 按作用域划分记忆：

- 用户记忆（User memory）—— 跨会话持久，按 `user_id` 键控。
- 会话记忆（Session memory）—— 在单个线程内持久。
- 智能体记忆（Agent memory）—— 每个智能体实例的状态。

每次写入选择一个作用域。检索可以跨作用域查询并为每个作用域指定权重。混用作用域而不加注意会导致“助手向 Alice 透露了关于 Bob 项目的信息”之类的事故。

### 该模式的常见问题

- 嵌入漂移（Embedding drift）。向量结果在最初的数百次查询看起来正确，但随着语料增长会退化。需定期对被高频使用的 top-N 记录重新嵌入。
- KV 模式蔓延（KV schema creep）。`(user_id, type, entity)` 看起来简单，直到每个团队都添加各自的 `type`。每季度审计 type 集合。
- 图膨胀（Graph explosion）。一个噪声较大的抽取器每条消息可能增加 50 条边。对每次 `add` 调用限制图写入；丢弃置信度低的边。

## 实现它

`code/main.py` 在 stdlib 中实现了三存储模式：

- `VectorStore` —— 采用基于 token 重叠的朴素相似度作为嵌入替代实现。
- `KVStore` —— 使用以 `(user_id, fact_type, entity)` 为键的字典。
- `GraphStore` —— 带类型的边（subject, relation, object, valid）。
- `Mem0` —— 顶层外观，包含 `add()`、`search()`、融合评分以及作用域感知检索。
- 一个对多用户、多会话对话的演示跟踪。

运行它：

```
python3 code/main.py
```

输出显示三条独立的召回路径以及融合后的 top-k。在 `main()` 顶部切换评分权重，观察排名变化。

## 使用它

- **Mem0 (Apache 2.0)** —— 可用于生产。可自托管（Postgres + Qdrant + Neo4j），或使用托管云服务。
- **Letta** —— 三层：核心/召回/归档；可自选向量和图后端。
- **Zep** —— 带有时序知识图谱与事实抽取的商业替代品。
- **自定义构建** —— 当你需要对抽取器（合规）或融合权重（语音代理中时效性占主导）有精确控制时。

## 练习

1. 用真实的嵌入模型替换玩具向量相似度（sentence-transformers、Ollama、OpenAI embeddings）。在合成的长对话上测量 recall@10。随着 1000 次写入，排名是否发生漂移？
2. 添加时序查询：`search(query, as_of=timestamp)`。只返回在该时间点或之前有效的记录。哪个存储需要最多工作量？
3. 实现冲突检测器：若传入事实与图边冲突，则将旧边标记为无效并记录两者。在“用户住在柏林”→“用户住在里斯本”上测试。
4. 将融合打分器扩展为包含 `user_feedback` 维度（对检索记录的点赞）。如何防止被操控（智能体只返回它已经被点赞的记录）？
5. 阅读 Mem0 文档（`docs.mem0.ai`）。将玩具实现迁移为对 `mem0` 客户端的调用。在相同的 20 个测试查询上比较检索质量。

## 关键词

| 术语 | 常说法 | 实际含义 |
|------|----------------|------------------------|
| 混合记忆 | “向量加图加 KV” | 三个并行存储，在写入时同时写入，检索时融合 |
| 事实抽取 | “记忆摄取” | 将文本分解为 (entity, relation, fact) 元组的 LLM 步骤 |
| 融合评分 | “相关性排序” | 相关性、重要性、时效性的加权和 |
| 作用域 | “记忆命名空间” | user / session / agent — 决定谁能看到什么 |
| Mem0g | “记忆图” | 带时序有效性的带类型边，用于关系查询 |
| 时间性失效 | “软删除” | 将被矛盾的边标记为无效；从不真正删除 |
| 嵌入漂移 | “检索退化” | 随语料增长向量质量退化；需定期重新嵌入 |
